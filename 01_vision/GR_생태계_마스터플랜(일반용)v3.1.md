# GR Framework 마스터플랜 v3.1
## "보안 컨설팅의 노하우를 확장 가능한 자산으로"

---

## 한 줄 요약

> **보안 컨설팅 회사의 전문 지식을 DB화하여, 하나의 Core로 여러 사업을 만드는 프로젝트**

---

## 1. 왜 이 프로젝트를 시작하는가?

### 문제: 컨설팅 비즈니스의 한계

```
일이 늘어나면 → 사람을 뽑아야 함 → 인건비 증가 → 수익률 정체
```

보안 컨설팅은 **"사람이 직접 하는 일"**입니다.
- 프로젝트 10개 하려면 → 10명 필요
- 프로젝트 100개 하려면 → 100명 필요

이것이 **선형 성장의 함정**입니다.

### 해결: 노하우를 "자산"으로 만들기

```
전문가의 머릿속 지식 → DB에 저장 → 여러 제품이 재사용 → 확장 가능한 사업
```

핵심 아이디어:
1. 컨설팅하면서 쌓은 **보안 노하우**를 체계적으로 정리
2. 이것을 **DB + Framework + 시각화(Atlas)**로 구축
3. 이 Core 위에 **여러 사업 아이템**을 올림

## 2. GR Core: 모든 것의 토대

### Core의 구성

```
┌─────────────────────────────────────────────────────────────┐
│                        GR CORE                              │
├───────────────────┬───────────────────┬─────────────────────┤
│      GR DB        │   GR Framework    │      GR Atlas       │
│                   │                   │                     │
│  "무엇이 있는가"   │  "어떻게 분류할까" │   "어떻게 보여줄까"  │
│                   │                   │                     │
│  • IT 제품 정보    │  • 3D 좌표 체계   │  • 지도처럼 보기     │
│  • 취약점(CVE)    │  • 관계 규칙      │  • 공격 흐름 시각화  │
│  • 공격 기법      │  • 보안 정책 규칙  │  • 직관적 이해      │
└───────────────────┴───────────────────┴─────────────────────┘
```

### 핵심 개념: IT 인프라 원자화

모든 IT 인프라를 **작은 조각(원자)**으로 분해하고, 각 조각의 **위치(좌표)**를 정합니다.

```
예시: 일반적인 웹 서비스 구조

[인터넷] → [방화벽] → [웹서버] → [API서버] → [데이터베이스]

GR Framework로 분류하면:

┌────────────┬─────────────┬──────────────┬────────────────┐
│   원자      │   Layer     │    Zone      │     역할       │
├────────────┼─────────────┼──────────────┼────────────────┤
│ 방화벽     │ L2(네트워크) │ Zone1(경계)  │ 외부 트래픽 차단│
│ Nginx      │ L2(네트워크) │ Zone1(경계)  │ 리버스 프록시   │
│ Spring API │ L7(앱)      │ Zone2(내부)  │ 비즈니스 로직   │
│ PostgreSQL │ L5(데이터)  │ Zone3(보호)  │ 데이터 저장     │
└────────────┴─────────────┴──────────────┴────────────────┘
```

### 왜 "좌표"가 중요한가?

**좌표가 정해지면, 보안 정책이 자동으로 따라옵니다.**

```
PostgreSQL이 Zone3(데이터 영역)에 있다면:
→ 자동으로 "암호화 필수" 정책 적용
→ 자동으로 "접근 제어 강화" 정책 적용
→ 자동으로 관련 CVE 취약점 알림
```

사람이 매번 "이 DB에는 어떤 보안이 필요하지?" 고민할 필요가 없습니다.
**위치만 정하면 Framework가 알려줍니다.**

## 3. MITRE ATT&CK과의 관계: WHERE + HOW

### 기존 보안 프레임워크의 한계

**MITRE ATT&CK**은 업계 표준 공격 기법 분류체계입니다.
하지만 MITRE는 **"어떻게(HOW)"** 공격하는지만 알려줍니다.

```
MITRE ATT&CK이 알려주는 것:
- T1190: 공개된 애플리케이션 취약점 공격
- T1059: 명령어 실행
- T1003: 인증 정보 탈취

→ "어떻게 공격하는지"는 알지만
→ "어디를 공격하는지"는 모름
```

### GR Framework의 역할: "WHERE"

```
┌─────────────────────────────────────────────────────────────┐
│                    완전한 보안 그림                          │
├─────────────────────────────┬───────────────────────────────┤
│       MITRE ATT&CK          │        GR Framework           │
│         (HOW)               │           (WHERE)             │
├─────────────────────────────┼───────────────────────────────┤
│  "이렇게 공격한다"           │  "여기가 공격당한다"           │
│                             │                               │
│  • 공격 기법 분류            │  • 인프라 위치 분류            │
│  • 공격 단계 정의            │  • 공격 대상 정의              │
│  • 방어 방법 제안            │  • 공격 경로 시각화            │
└─────────────────────────────┴───────────────────────────────┘

                    ↓ 결합하면 ↓

        "어디에서 + 어떻게" = 실행 가능한 보안 전략
```

### 실제 예시: Log4Shell 취약점

```
MITRE만 보면:
  "T1190 기법으로 공격할 수 있다" (끝)

GR + MITRE를 보면:
  1. [Zone1 → Zone2] 웹서버에서 Log4j 취약점 공격 (T1190)
  2. [Zone2 → Zone3] 내부 네트워크로 이동, DB 접근 시도 (T1003)
  3. 구체적 대응: Zone2에 WAF 강화, Zone3 접근 제어 강화
```

**GR Framework는 MITRE와 경쟁하지 않습니다. 보완합니다.**

## 4. GR Atlas: 보안의 지도

### 개념

**GR Atlas**는 인프라를 **지도처럼** 보여주는 시각화 기능입니다.

```
일반 지도                          GR Atlas
─────────────                      ─────────
• 지형, 도로, 건물                 • Layer, Zone, 서버
• 교통 흐름                        • 데이터 흐름
• 위험 지역 표시                   • 취약점 위치 표시
• 최적 경로 안내                   • 공격 경로 시각화
```

### 왜 "지도"인가?

보안 담당자가 가장 어려워하는 것:
- "우리 인프라 전체가 어떻게 생겼지?"
- "공격자가 들어오면 어디로 갈 수 있지?"
- "어디가 가장 위험하지?"

**Atlas는 이것을 한눈에 보여줍니다.**

```
┌─────────────────────────────────────────────────────────────┐
│                      GR Atlas 화면                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [인터넷]                                                  │
│      │                                                      │
│      ▼                                                      │
│   ┌──────┐     ┌──────┐     ┌──────┐                       │
│   │ WAF  │────▶│ API  │────▶│  DB  │                       │
│   │ ⚠️   │     │ 🔴   │     │ ✅   │                       │
│   └──────┘     └──────┘     └──────┘                       │
│   Zone 1       Zone 2       Zone 3                         │
│                                                             │
│   ⚠️ 설정 미흡  🔴 CVE 발견  ✅ 정상                        │
│                                                             │
│   [공격 경로] Zone1 → Zone2 → Zone3 (위험도: 높음)          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 5. Core 위에 올라가는 사업들

GR Core(DB + Framework + Atlas)는 **직접 돈을 벌지 않습니다.**
Core 위에 올라가는 **제품들**이 돈을 벌고, Phase 3 이후에는 **API Economy 모델**(사용량 기반 과금)로 Core 자체도 수익원이 됩니다.

### 사업 아이템 4가지

```
┌─────────────────────────────────────────────────────────────┐
│                     수익 창출 제품들                         │
├───────────────┬───────────────┬───────────────┬─────────────┤
│  자동화 진단   │  보안 솔루션   │    GR IaC    │   보안 교육  │
│               │               │              │             │
│ 컨설팅 자동화  │ 탐지/대응     │ 보안 내재    │  실습 환경   │
│               │               │ 인프라 코드  │  자동 구축   │
└───────────────┴───────────────┴───────────────┴─────────────┘
                              ▲
                              │
                    ┌─────────┴─────────┐
                    │     GR CORE       │
                    │  DB + Framework   │
                    │     + Atlas       │
                    └───────────────────┘
```

### 1) 자동화 진단 도구

**현재**: 컨설턴트가 직접 인프라 점검 → 보고서 작성 (2-4주)
**GR 적용 후**: 시스템이 자동 스캔 → 즉시 결과 (1-2일)

```
입력: 고객사 인프라 정보
        ↓
    [GR Framework]
    • 각 구성요소 좌표 자동 할당
    • 보안 정책 자동 적용
    • 취약점 자동 매칭
        ↓
출력: 상세 진단 보고서 + Atlas 시각화
```

**수익**: 기존 컨설팅 대비 10배 많은 고객 처리 가능

### 2) 보안 솔루션

진단 도구로 **문제를 발견**하면, 솔루션으로 **문제를 해결**합니다.

```
진단 결과: "Zone2의 API 서버에서 비정상 접근 탐지"
        ↓
    [GR 기반 솔루션]
    • 해당 좌표(Zone2, L7)에 맞는 대응 규칙 적용
    • 자동 차단 또는 알림
        ↓
결과: 위협 대응 완료
```

**차별점**: 다른 솔루션은 "이상 행위"만 탐지. GR 솔루션은 **"어디서 왜"**까지 설명.

### 3) GR IaC (Infrastructure as Code)

**IaC**: 인프라를 코드로 정의하여 자동 구축하는 방식

일반 IaC의 문제:
- 인프라는 만들어지지만
- 보안 설정은 별도로 해야 함
- 실수하면 취약점 발생

**GR IaC의 차별점**:
```
입력: "3-tier 웹 서비스 만들어줘"
        ↓
    [GR IaC]
    • 인프라 코드 생성 (Terraform)
    • 각 구성요소에 GR 좌표 자동 할당
    • 좌표에 맞는 보안 설정 자동 포함
        ↓
출력: 보안이 내장된 인프라 코드
```

"만들면서 동시에 보안이 적용되는" 인프라

### 4) 보안 교육 (GR Edu)

보안 교육의 가장 큰 문제: **실습 환경 부족**

- 이론만 배우면 실전에서 못 씀
- 실습 환경 만드는데 시간/비용 많이 듦
- CVE 나올 때마다 새 실습 환경 필요

**GR Edu의 해결책**:
```
입력: "CVE-2021-44228 (Log4Shell) 실습하고 싶어"
        ↓
    [GR IaC Lab]
    • 해당 CVE가 발생하는 환경 자동 구축
    • 취약한 버전 + 공격 시나리오 준비
    • 방어 실습까지 포함
        ↓
출력: 즉시 사용 가능한 실습 환경
```

## 6. 시너지: 순환하는 사업 구조

4개 사업은 **독립적이지 않고 서로 연결**됩니다.

```
         ┌──────────────────┐
         │   자동화 진단     │
         │  (문제 발견)      │
         └────────┬─────────┘
                  │
    ┌─────────────┼─────────────┐
    ▼             │             ▼
┌───────┐         │         ┌───────┐
│ 보안  │         │         │  IaC  │
│ 교육  │◀────────┼────────▶│       │
│       │         │         │       │
└───┬───┘         │         └───┬───┘
    │             ▼             │
    │      ┌───────────┐        │
    └─────▶│ 보안 솔루션 │◀──────┘
           │ (문제 해결) │
           └───────────┘
```

**순환 관계**:
1. **진단**으로 문제 발견 → **솔루션**이 해결
2. **솔루션** 도입할 인프라 → **IaC**로 자동 구축
3. **IaC**로 만든 환경 → **교육** 실습에 활용
4. **교육**받은 인력 → **진단** 도구 사용

**하나가 팔리면 다른 것도 자연스럽게 연결됩니다.**

## 7. 기술 구조 (간략)

### 3D 좌표 체계

모든 IT 구성요소를 **3개의 축**으로 분류합니다.

```
                    Layer (깊이)
                        │
                        │   L7: 애플리케이션
                        │   L6: 런타임
                        │   L5: 데이터
                        │   L4: 플랫폼
                        │   L3: 컴퓨팅
                        │   L2: 네트워크
                        │   L1: 물리
                        │   L0: 외부 서비스
                        │
    Zone (신뢰도)───────┼───────── Tag (기능)
                        │
    Zone0: 비신뢰                 M: 관리
    Zone1: 경계                   N: 네트워크
    Zone2: 내부                   S: 보안
    Zone3: 데이터                 A: 애플리케이션
    Zone4: 관리                   D: 데이터
    Zone5: 엔드포인트             R: 런타임
                                  C: 컴퓨팅
                                  P: 플랫폼
                                  T: 모니터링·관측
                                  I: 연동·통합
```

**예시**: PostgreSQL을 캐시로 쓰는 경우
- Layer: L5 (데이터 서비스)
- Zone: Zone2 (애플리케이션 영역)
- Tag: D3.1 (캐시)
- **좌표**: (L5, Zone2, D3.1)

### 하이브리드 DB 구조

```
┌─────────────────────────────────────────────────────────────┐
│                    GR DB Architecture                        │
├───────────────────┬───────────────────┬─────────────────────┤
│    PostgreSQL     │      Neo4j        │      Pinecone       │
│                   │                   │                     │
│  "사실 저장"       │  "관계 저장"       │  "의미 저장"         │
│                   │                   │                     │
│  • 제품 정보       │  • 공격 경로       │  • 유사 제품 검색    │
│  • 버전, 라이선스  │  • 의존성 관계     │  • AI 검색 지원     │
│  • CVE 데이터     │  • Zone간 연결     │  • 자연어 질의      │
└───────────────────┴───────────────────┴─────────────────────┘
```

## 9. 왜 지금인가?

### 2025년 AI 시대의 기회

```
과거: 데이터 구축 = 사람이 직접 = 비용 막대
현재: 데이터 구축 = AI 자동화 = 비용 현실적
```

- LLM이 제품 문서를 읽고 분류 가능
- 크롤링 + AI로 대량 데이터 구축 가능
- 소규모 팀으로도 대규모 DB 구축 가능

### AI의 2단계 역할

GR Framework에서 AI(LLM)는 **2단계로 활용**됩니다.

```
┌─────────────────────────────────────────────────────────────┐
│                     AI 역할 2단계                            │
├─────────────────────────────┬───────────────────────────────┤
│      1단계: 구축            │       2단계: 운영              │
├─────────────────────────────┼───────────────────────────────┤
│  • CVE 수집 및 분석          │  • 복잡한 질의 처리 (20%)     │
│  • 제품-취약점 자동 매핑     │  • 자연어 검색 지원           │
│  • 문서 읽고 좌표 분류       │  • 맞춤형 보고서 생성         │
│  • 대량 데이터 정제          │  • 공격 경로 추론             │
└─────────────────────────────┴───────────────────────────────┘
```

**80/20 원칙**: 운영 시 80%는 DB 직접 조회, 20%는 AI-Assisted 추론
→ 비용 효율적이면서도 복잡한 분석 가능

### 보안 컨설팅 회사가 하기에 적합한 이유

1. **실무 데이터 접근성**: 매일 고객사 인프라를 봄
2. **검증 기회**: 컨설팅하면서 Framework 테스트 가능
3. **즉시 사용자**: 본인 회사가 첫 번째 고객
4. **도메인 전문성**: 외부 개발사보다 보안을 잘 앎

---

## 10. 핵심 메시지

### 이 프로젝트가 아닌 것

- ❌ 또 하나의 보안 제품
- ❌ MITRE ATT&CK의 경쟁자
- ❌ 단기 수익 프로젝트

### 이 프로젝트가 맞는 것

- ✅ 모든 보안 제품의 **토대**
- ✅ MITRE ATT&CK의 **보완재** (WHERE + HOW)
- ✅ 컨설팅 노하우의 **자산화**
- ✅ **확장 가능한** 사업 모델

---

## 결론

> **"우리는 제품을 만들지 않습니다. 모든 보안 제품이 서 있을 토대를 만듭니다."**

GR Framework는 보안 컨설팅 회사가 **"일 ↔ 인력"의 선형 성장**에서 벗어나,
**"Core ↔ 여러 제품"의 확장 가능한 성장**으로 전환하기 위한 프로젝트입니다.

```
Before: 일 10개 = 사람 10명 = 수익 10
After:  Core 1개 = 제품 4개 = 수익 40+
```

이것이 GR Framework의 본질입니다.
